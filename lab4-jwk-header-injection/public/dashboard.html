Attack Instructions — JWK Header Injection

The server supports the jwk parameter in JWT headers but fails to check if the key came from a trusted source. You can embed your own public key and sign with your private key.

1 Login and intercept in Burp Suite
Login with wiener / peter. Send the post-login request to Burp Repeater.
2 Generate a new RSA key pair
In Burp → JWT Editor Keys tab → click New RSA Key → click Generate → click OK.
3 Modify the JWT payload
In Burp Repeater, switch to the JSON Web Token tab. Change the payload: 
// Change the "sub" claim:
{
  "sub": "admin",
  "role": "admin",
  "iat": ...,
  "exp": ...
}
4 Use the Embedded JWK attack
  1. At the bottom of the JSON Web Token tab, click Attack
  2. Select Embedded JWK
  3. Choose your newly generated RSA key
  4. Click OK
This embeds your public key in the header and signs with your private key: 
// The JWT header will now contain:
{
  "alg": "RS256",
  "typ": "JWT",
  "kid": "your-key-id",
  "jwk": {
    "kty": "RSA",
    "n": "...",     ← your public key modulus
    "e": "AQAB",
    "kid": "your-key-id"
  }
}
5 Access the vulnerable admin panel
The server will use your embedded JWK to verify the token — verification passes because you signed with the matching private key. Access /api/vulnerable-admin (or click the button below) to solve the lab.

Why This Works

The server's vulnerable verification logic:

// VULNERABLE — trusts attacker-supplied key
const header = decodeHeader(token);
if (header.jwk) {
    // Uses the JWK from the token header to verify!
    const key = crypto.createPublicKey({ key: header.jwk, format: 'jwk' });
    decoded = jwt.verify(token, key);  // ← attacker controls the key!
}

The attacker signs with their own private key and embeds the matching public key — so verification always passes.
